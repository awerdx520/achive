# AGENTS.md - Emacs Lisp 项目规范

本文档为 achive 项目（一个 Emacs Lisp 插件）的 AI 助手提供工作规范和指导。

## 项目概述

achive 是一个基于新浪 API 的 Emacs 插件，用于实时显示 A 股、港股和美股行情。项目主要文件包括：
- `achive.el`：主程序文件
- `test-achive.el`：单元测试文件
- `Makefile`：构建和测试脚本

## 交互要求

### 语言要求
- **始终使用中文回复用户的所有问题**：包括代码解释、错误消息和所有交互
- **代码注释也使用中文编写**：所有代码注释、文档字符串和文档都应使用中文
- **思考过程用中文表述**：在分析问题时，内部思考过程也应使用中文

### 交互风格
- 请使用专业且严谨的语气进行回答
- 当用户提问过于宽泛时，请主动追问具体细节
- 如果发现代码存在潜在风险，请主动提出警告

### 上下文管理
- 每次对话请参考最近 5 条历史记录
- 对于大型文件，请优先关注函数级别的修改，而非全局重构

## 代码规范

### Emacs Lisp 编码风格
1. **命名约定**：
   - 变量和函数名使用小写字母，单词间用连字符分隔（如 `achive-get-market`）
   - 自定义组使用 `achive-` 前缀（如 `achive-face-up`）
   - 常量使用 `achive-` 前缀（如 `achive-api`）

2. **文档字符串**：
   - 所有函数和变量都应提供中文文档字符串
   - 文档字符串应清晰描述功能、参数和返回值
   - 示例：
     ```elisp
     (defun achive-get-market (code)
       "根据股票代码识别市场。
       CODE: 股票代码，如 sh000001, hk00700, usAAPL。
       返回市场标识：a-share, hk, us 或 nil。"
       ...)
     ```

3. **代码组织**：
   - 使用 `;;;` 分隔主要部分（如 `;;;; Requirements`）
   - 使用 `;;` 进行常规注释
   - 使用 `;` 进行行内注释

### 安全最佳实践
- **禁止在代码中硬编码 API Keys 或密码**：所有敏感信息应通过配置变量提供
- **禁止输出任何包含个人身份信息（PII）的内容**：包括日志、错误消息和用户数据
- **输入验证**：对所有用户输入和 API 响应进行验证，防止注入攻击
- **错误处理**：使用 `condition-case` 妥善处理可能失败的函数调用

## 构建与测试

### 常用命令
项目使用 Makefile 进行构建和管理。以下是常用命令：

```bash
# 编译并运行测试（默认）
make all

# 编译所有 .el 文件
make compile

# 运行所有单元测试
make test

# 交互式运行测试
make test-interactive

# 检查语法
make check-syntax

# 清理编译文件
make clean

# 安装到 Emacs 目录
make install

# 创建发布包
make dist

# 启动开发环境
make dev
```

### 测试规范
1. **测试框架**：使用 Emacs 自带的 ERT（Emacs Lisp Regression Testing）框架
2. **测试文件**：`test-achive.el` 包含所有测试用例
3. **测试运行**：确保所有测试在修改后通过
4. **测试编写**：
   - 使用 `ert-deftest` 定义测试用例
   - 测试名称应以 `test-` 开头
   - 每个测试应有清晰的中文描述

### 代码质量检查
- **语法检查**：运行 `make check-syntax` 确保代码语法正确
- **编译检查**：确保所有 `.el` 文件都能成功编译为 `.elc`
- **警告处理**：修复所有编译警告，特别是关于未声明变量的警告

## 开发工作流

### 修改现有代码
1. **理解上下文**：在修改前，先阅读相关函数及其调用关系
2. **遵循现有模式**：查看相邻代码的编写风格和组织方式
3. **测试驱动**：先编写或修改测试，然后实现功能
4. **小步提交**：每次提交应包含逻辑完整的变更

### 添加新功能
1. **检查依赖**：确认项目是否已使用相关库，避免引入新依赖
2. **代码审查**：查看类似功能的实现方式
3. **文档更新**：更新相关文档和注释
4. **测试覆盖**：为新功能添加相应的测试用例

### 错误修复
1. **重现问题**：首先理解并重现报告的错误
2. **根因分析**：定位问题根源，而不仅仅是修复表面症状
3. **回归测试**：确保修复不会引入新的问题
4. **示例更新**：如果有相关示例代码，确保其仍然有效

## 版本控制

### Git 规范
1. **提交消息**：
   - 使用中文提交消息
   - 简明描述变更内容和原因
   - 格式：`类型: 简短描述`
   - 示例：`功能: 添加港股市场支持`

2. **分支管理**：
   - `main` 分支用于稳定版本
   - 功能开发在特性分支进行
   - 修复在修复分支进行

3. **提交前检查**：
   - 运行 `make test` 确保测试通过
   - 运行 `make check-syntax` 确保语法正确
   - 检查代码风格一致性

## 特殊注意事项

### 多市场支持
achive 支持 A 股、港股和美股市场，需要注意：
- 不同市场的交易时间不同
- 美股代码需要转换为新浪 API 的特殊格式（`usAAPL` → `gb_aapl`）
- 不同市场的数据字段可能有所差异

### 实时数据更新
- 使用定时器自动刷新数据
- 仅在交易时间自动更新
- 处理网络请求失败的情况

### 用户配置
- 尊重用户的自定义设置（如 `achive-index-list`、`achive-stock-list`）
- 提供合理的默认值
- 配置变更向后兼容
